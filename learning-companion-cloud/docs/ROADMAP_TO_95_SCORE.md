# 学习陪跑 Agent 达到 9.5 分的完整规划

> 当前综合评分约 8.2–8.5。要达到 9.5，不是再堆几个按钮，而是把系统从“能跑通的受控学习 Agent”升级为“教材扎实、题目高质、Agent 可解释、评测可信、真实使用稳定”的学习产品和 AI 测试开发项目。

## 1. 9.5 分定义

9.5 分不是内部脚本全绿，而是同时满足以下五类标准：

| 维度 | 9.5 分标准 |
|---|---|
| 产品可用 | 家长一句话下目标，系统自动生成合理计划；孩子按流程学习 7 天不需要开发者解释 |
| 学科有效 | 五上语文、数学、英语知识点覆盖完整，题目能诊断 95+ 目标差距 |
| Agent 可信 | 有受控 Tool Loop、RAG 证据、状态机、记忆、反思补漏、可观测 trace |
| 测评可信 | 有真实红例、轨迹评估、LLM-as-Judge、红队 case、CI 门禁，不自嗨 100% |
| UI 易用 | 孩子端聚焦、家长端看结论、管理端少维护，视觉不乱、不低幼、不堆信息 |

## 2. 当前差距总览

| 差距 | 当前状态 | 到 9.5 必须补齐 |
|---|---|---|
| 教材知识库 | 有种子资料和本地导入能力，但不是完整结构化知识库 | 三科五上全章节、知识点、题型、资料来源结构化 |
| RAG | 关键词 + 轻量词向量混合检索 | embedding 向量库 + rerank + 引用证据 + 覆盖率看板 |
| 题目质量 | 本地兜底可用，但偏基础 | 分层题、错因题、变式题、阅读题、步骤诊断题、听写/默写题 |
| Agent Loop | 有受控 Tool Loop | 扩展为 Planner/Quiz/Review/Report 多节点受控 Runtime |
| 学习记忆 | 掌握度和错因有雏形 | 1/3/7 天复习、连续失败升级、连续通过降频 |
| Eval | 89 条左右，Known Gap 已有 | 150+ case，含真实教材、轨迹级评估、LLM-as-Judge、红队集 |
| UI | 三端结构可用 | 儿童聚焦体验、家长老师式结论、管理端质量驾驶舱 |
| 真实验证 | 自动化测试充分 | 至少 7 天模拟学习数据 + 1 次完整人工验收脚本 |

## 3. 总体路线

```text
阶段 1：知识库补齐
```

## 4. 阶段 1：教材知识库补齐

### 4.1 目标

把五年级上册语文、数学、英语从“资料片段”升级为“可检索、可出题、可评估”的结构化知识库。

### 4.2 语文知识库

必须覆盖：

- 课文目录。
- 每课生字、会认字、会写字。
- 重点词语。
- 课后题。
- 单元语文园地。
- 日积月累。
- 习作要求。
- 阅读理解能力点。

建议数据结构：

```json
{
  "subject": "语文",
  "unit": "第一单元",
  "lesson": "白鹭",
  "sections": [
    {"type": "text", "title": "课文正文", "content": "..."},
    {"type": "dictation", "items": ["精巧", "适宜", "色素"]},
    {"type": "understanding", "points": ["借物抒情", "关键句理解"]},
    {"type": "accumulation", "items": ["日积月累内容"]}
  ]
}
```

### 4.3 数学知识库

必须覆盖：

- 单元目录。
- 每节概念。
- 例题。
- 练习题型。
- 公式/方法。
- 易错点。
- 单元测试目标。

重点能力：

- 小数乘法/除法。
- 位置。
- 可能性。
- 多边形面积。
- 方程。
- 植树问题。
- 应用题建模。

### 4.4 英语知识库

必须覆盖：

- Unit 目录。
- 单词表。
- 重点句型。
- Story/Dialog。
- 听写材料。
- 中译英材料。
- 阅读理解材料。
- 音频文件线索。

### 4.5 实施任务

| 编号 | 任务 | 产物 |
|---|---|---|
| K1 | 新增 `backend/knowledge_schema.py` | 统一三科知识点结构 |

| K3 | 新增 `scripts/import_textbook_knowledge.py` | PDF/文本转知识库导入脚本 |

| K5 | 管理端新增覆盖率面板 | 显示每科每单元缺口 |

### 4.6 验收标准

- 三科每科至少 6 个单元结构化。
- 每科至少 30 个知识点。
- 语文至少 100 个字词/积累点。
- 数学至少 60 个概念/例题/易错点。
- 英语至少 100 个单词/句型/听写点。
- 任意选择一个知识点，能生成任务和测验。

### 4.7 预期提升

教材/RAG 深度：`6.8 -> 8.2`

## 5. 阶段 2：RAG 升级

### 5.1 目标

从轻量混合检索升级为更可靠的教材证据检索。

### 5.2 技术方案

优先本地可运行：

1. 保留关键词检索，保证可解释。
2. 增加 `sentence-transformers` 或 OpenAI-compatible embedding。
3. 使用本地 SQLite 存 embedding 向量，或引入 FAISS。
4. 增加 rerank：章节匹配、学科匹配、知识点匹配、教材来源优先。
5. 任务、题目、报告都返回 `source_refs`。

### 5.3 实施任务

| 编号 | 任务 | 产物 |
|---|---|---|
| R1 | 新增 `backend/rag_engine.py` | 统一检索入口 |
| R2 | 新增 `material_embeddings` 表 | 存储向量和 chunk 关系 |
| R3 | 新增 embedding 生成脚本 | 支持本地/远程 embedding |
| R4 | 改造 `search_material_chunks` | keyword + vector + rerank |
| R5 | 改造题目生成 | 每题必须绑定资料证据 |

### 5.4 验收标准

- 真实中文 query 命中对应课文/例题/单词。
- 每道题有 `source_ref`。
- 每个任务至少 1 条资料证据。
- Known Gap 中至少 2 条 RAG 红例被修复，同时保留新的边界红例。
- RAG Eval 不允许 100% 全绿，必须说明边界。

### 5.5 预期提升

RAG 可信度：`7.2 -> 8.8`

## 6. 阶段 3：题目引擎升级

### 6.1 目标

让小测从“能出题”变成“能诊断是否能考 95+”。

### 6.2 语文题型

必须新增/强化：

- 生字听写。
- 拼音读音。
- 词语解释。
- 组词造句。
- 课文关键句理解。
- 日积月累默写。
- 阅读理解。
- 小练笔 rubric。

### 6.3 数学题型

必须新增/强化：

- 精确计算。
- 竖式过程。
- 概念判断。
- 填空。
- 应用题。
- 步骤说明。
- 错因归类。
- 同类变式。

### 6.4 英语题型

必须新增/强化：

- 单词默写。
- 中英互译。
- 句型填空。
- 连词成句。
- 听写。
- 阅读理解。
- 造句。
- 错词复写。

### 6.5 题目质量门禁

每套题必须检查：

- 不泄露答案。
- 不出现题干和选项互相提示。
- 不出超纲题。
- 不只有机械题。
- 至少包含基础、理解、应用。
- 每题有评分规则。
- 开放题不能用“回答不为空即可”。

### 6.7 验收标准

- 每科至少 8 类题型。
- 每科至少 30 条题目 Eval case。

- 数学应用题能输出错因分类。
- 语文听写和英语默写能严格判分。

### 6.8 预期提升

题目质量：`8.0 -> 9.0`

## 7. 阶段 4：Agent Runtime 升级

### 7.1 目标

从“单点受控 Tool Loop”升级为“多节点受控 Agent Runtime”。

### 7.2 Runtime 设计

```text
Plan Node
  -> Retrieve Node
  -> Decide Task Node
  -> Guide Node
  -> Quiz Node
  -> Grade Node
  -> Review Node
  -> Report Node
  -> Reflect Node
```

每个 Node 都要记录：

- 输入。
- 工具选择。
- 工具校验。
- 观察结果。
- 决策理由。
- 输出。
- 失败兜底。
- trace。

### 7.3 受控原则

- 模型不能直接写数据库，必须通过工具。
- 写工具必须 schema 校验。
- 儿童场景不允许模型自由越权。
- Agent 可以建议，但关键任务状态由规则状态机控制。

### 7.4 实施任务

| 编号 | 任务 | 产物 |
|---|---|---|
| A1 | 扩展 `agent_runtime.py` | 多步 node runtime |
| A2 | 改造 Planner/Quiz/Review/Report | 都走 Runtime |
| A3 | `agent_trace_steps` 增加 node 信息 | 可观测轨迹 |
| A4 | 管理端优化决策日志 | 默认显示结论，展开看轨迹 |
| A5 | 新增 Agent trace Eval | 评估工具选择和步骤质量 |

### 7.5 验收标准

- 至少 4 条链路经过 Runtime：计划、卡住、出题、报告。
- 每条 Agent run 至少 2 个 trace step。
- 工具调用失败能 fallback。
- 管理端能看清“为什么这么安排”。

### 7.6 预期提升

Agent 架构：`7.8 -> 8.8`

## 9. 阶段 6：UI/体验升级

### 9.1 目标

让系统真的适合 11 岁孩子和家长使用，而不是开发者自测工具。

### 9.2 孩子端

升级点：

- 当前任务更突出。
- 计时更醒目。
- 卡住辅导变成步骤卡片。
- 小测反馈更像闯关，不像表单。
- 完成后明确告诉“能不能进入下一项”。
- 多任务队列只展示必要信息。

### 9.3 家长端

升级点：

- 默认只看三句话：
  - 今天完成了吗？
  - 风险在哪里？
  - 明天我做什么？
- 把 Agent 日志、详细数据折叠。
- 增加“10 分钟家长介入动作”。

### 9.4 管理端

升级点：

- 默认只保留三件事：
  - 输入目标。
  - 生成任务。
  - 看资料覆盖。
- Agent 决策日志折叠成“质量证据”。
- 复杂调试信息默认隐藏。

### 9.5 实施任务

| 编号 | 任务 | 产物 |
|---|---|---|
| U1 | 孩子端任务卡重构 | 单焦点学习界面 |
| U2 | 小测界面重构 | 闯关式反馈 |
| U3 | 卡住辅导卡片化 | 一步一步操作 |
| U4 | 家长端三句话看板 | 结论优先 |
| U5 | 管理端质量驾驶舱 | 少维护、看覆盖 |

### 9.6 验收标准

- 人工打开三端，5 分钟内能完成一次完整流程。
- 孩子端不需要看英文标题理解按钮。
- 家长端不需要读日志就知道今天情况。
- UI Rubric ≥ 9.5。
- 截图审计无错位、无脏文本、无控制台错误。

### 9.7 预期提升

UI 易用性：`8.0 -> 9.0`

## 12. 优先级排序

如果只能按价值排序，建议顺序如下：

1. 教材知识库补齐。
2. 题目引擎升级。
3. Agent Eval 扩展到 200 case。
4. RAG embedding + rerank。
5. 7 天闭环模拟。
6. UI 体验精修。
7. 多节点 Runtime 扩展。

理由：

- 对孩子学习效果最关键的是知识库和题目。
- 对面试最关键的是 Eval 和 trace。
- 对家长使用最关键的是 7 天闭环和 UI。
- 完整自主 Agent 不是第一优先级，受控、可解释、可评估更重要。

## 15. 风险与取舍

| 风险 | 处理 |
|---|---|
| 教材版权和资料来源 | 优先用户提供 PDF/资料；平台资料只做学习索引，不直接分发版权内容 |
| AI 不稳定 | 所有核心链路保留规则兜底 |
| 全自主 Agent 不适合儿童 | 采用受控 Runtime，不让模型越权 |
| 题目质量难自动判断 | 规则检测 + Eval case + LLM-as-Judge + 人工抽检 |

# 11 岁孩子暑假自主学习陪跑系统：系统级规划与验收方案

> 本文档替代旧版 Agent 升级规划，作为后续开发、测试、复盘、面试讲解的唯一主规划。
> 当前系统定位必须诚实：这是“受控学习 Agent”，不是完全自主 Agent。它的核心价值是把家长目标、教材资料、孩子执行过程、AI 辅导、测验批改、复习补漏和家长结论串成闭环。

## 1. 一句话目标

围绕“武汉小学新五年级上册语文、数学、英语，目标考试 95 分以上”，构建一个可本地运行、可接入 OpenAI-compatible 模型、可被测试和评估的学习陪跑系统：

- 家长只输入学习目标、资料范围和每天可用时间。
- 系统自动拆解长期计划、每日任务、学习步骤和复习节奏。
- 孩子端只聚焦当前任务，支持开始、暂停、卡住求助、完成检查、进入下一任务。
- Agent 根据教材知识库和学习表现生成测验、批改、诊断错因、安排补漏。
- 家长端只看结论：今天学得怎么样、哪里有风险、明天怎么安排、是否需要家长介入。
- 管理端只做少量维护：导入资料、触发计划、查看 Agent 决策证据和质量状态。

## 2. 产品边界

### 2.1 必须覆盖

| 模块 | 必须能力 | 验收标准 |
|---|---|---|
| 三端页面 | 孩子端、家长端、管理端角色分离 | 三端关键按钮可点击，核心链路可完成 |
| 学习规划 | 根据自然语言目标生成学习来源和每日任务 | 能从“每日一课/一小节”生成今日任务 |
| 教材知识库 | 语文、数学、英语五上资料纳入检索 | 任务和测验能引用对应知识点 |
| 学习过程 | 开始、暂停、计时、卡住、完成检查、下一个任务 | 状态机不乱跳，重复点击不写脏数据 |
| 卡住辅导 | 针对具体卡点给解题思路，不直接喂答案 | 没有卡点时引导孩子补充，不能输出废话 |
| 小测验 | 按学科和知识点生成题目 | 题目不能泄露答案，题型适配学科 |
| 批改诊断 | 判分、错因、掌握度、补漏建议 | 低分自动进入复习/补漏策略 |
| 家长报告 | 生成日结论、风险、下一步建议 | 家长不需要看流水账即可判断是否干预 |
| Agent 可观测 | 记录 run 和 trace steps | 能解释 Agent 为什么这么安排 |
| 测评框架 | 有功能测试、UI 测试、Agent Eval、质量门禁 | 本地和 CI 都能跑，Known Gap 单独标识 |

### 2.2 不做或谨慎做

- 不让模型完全自由规划孩子学习路径，关键动作必须受教材、目标和规则约束。
- 不把“AI 生成的内容”默认当教材事实，必须优先使用本地资料/RAG 命中内容。
- 不让孩子端出现管理端、家长端入口，避免角色混淆。
- 不上传 `.env`、API Key、`learning.db` 等个人数据。
- 不承诺替代老师和考试命题，只做家庭自主学习陪跑和诊断。

## 3. 用户角色与核心流程

### 3.1 孩子端

孩子每天只需要按顺序完成任务：

1. 打开孩子端，看到“当前最该做的一件事”。
2. 点击“开始”，计时启动，状态变为学习中。
3. 学习过程中可点击“暂停”，计时停止，再点“开始”继续。
4. 如果不会，点击“我卡住了”，输入具体卡点，Agent 给分步提示。
5. 学会后继续当前任务，不应重新开始已进入检查阶段的任务。
6. 做完后点击“我做完了，开始检查”，进入测验。
7. 提交答案后获得得分、错因和是否通过。
8. 通过后进入下一个任务；未通过则进入补漏或二次练习。

### 3.2 家长端

家长每天只回答三个问题：

1. 今天完成了吗？
2. 有没有明显风险？
3. 明天需要我做什么？

页面只应突出结论、风险、建议，细节通过折叠区域查看。

### 3.3 管理端

管理端负责系统准备和证据查看：

1. 输入自然语言计划，例如“寒假作业本每日一小节，语文书每日一篇课文，英语每天听写 10 个单词”。
2. 导入或维护教材资料。
3. 生成学习计划和今日任务。
4. 查看资料覆盖度、Agent 决策日志、测评结果。
5. 必要时手动补录资料或重新生成任务。

## 4. Agent 架构

### 4.1 当前定位

当前采用“受控 Agent Runtime”：

```text
用户目标/学习资料/历史表现
  -> 规则编排
  -> RAG 检索
  -> 受控工具调用契约校验
  -> LLM JSON 生成或本地规则兜底
  -> 状态机落库
  -> 测验/批改/报告
  -> trace 可观测
```

它不是完整 ReAct 自主循环，因为儿童学习场景不适合让模型自由选择任意动作。更合理的设计是“人设目标、规则控边界、AI 做增强、测试兜底”。

### 4.2 核心 Agent 模块

| Agent | 输入 | 过程 | 输出 |
|---|---|---|---|
| Planner Agent | 家长目标、教材资料、时间约束 | 拆长期目标、优先级、任务来源 | 学习计划、资料覆盖缺口 |
| Daily Task Agent | 学习计划、日期、历史掌握度 | 安排新课、复习、补漏比例 | 今日任务队列 |
| Learning Guide Agent | 当前任务、知识点、学科 | 生成怎么学、怎么练、怎么检查 | 孩子端学习步骤 |
| Stuck Assist Agent | 当前任务、卡点描述、资料片段 | 分类卡点并给提示 | 不直接给答案的分步辅导 |
| Quiz Agent | 任务、知识点、题型策略 | 生成小测并做防泄露检查 | 题目、标准答案、评分规则 |
| Grading Agent | 题目、答案、评分规则 | 本地精确判分 + AI 开放题判断 | 得分、错因、掌握度 |
| Review Agent | 错因、掌握度、间隔复习规则 | 生成补漏任务和复习节奏 | 明日补漏/1-3-7 天复习 |
| Parent Report Agent | 今日任务、测验、错因、时间 | 压缩成家长结论 | 日报、风险、介入建议 |

### 4.3 工具契约

所有可执行动作都应有工具契约：

- `generate_daily_tasks`
- `generate_quiz`
- `grade_quiz`
- `assist_stuck`
- `write_memory`
- `search_materials`
- `create_review_task`
- `generate_parent_report`

每个工具必须声明：

- 参数 schema
- 是否有副作用
- 是否幂等
- 权限边界
- 输出 schema
- trace 记录

## 5. 知识库与 RAG 规划

### 5.1 资料范围

| 学科 | 必须资料 | 关键知识点 |
|---|---|---|
| 语文 | 五上课文、生字词、语文园地、日积月累、习作要求 | 字词、朗读、理解、表达、积累、听写 |
| 数学 | 五上教材、例题、练习、单元知识结构 | 小数乘除法、位置、多边形面积、方程、统计、植树问题 |
| 英语 | 五上教材、单词表、句型、听写资料 | 单词默写、词义、句型、阅读、听说 |

### 5.2 检索策略

当前可用关键词 RAG，下一阶段建议升级为混合检索：

1. 关键词检索：稳定、可解释，适合小规模教材。
2. 中文分词：提升语文和数学概念召回。
3. 向量检索：支持同义表达和开放问题。
4. rerank：把教材命中、章节匹配、年级匹配作为排序因子。
5. 引用证据：每个任务和测验尽量带资料来源。

### 5.3 资料质量门禁

- 不允许只用模型常识替代教材资料。
- 资料缺失时，管理端必须提示“覆盖不足”。
- Known Gap 要保留，不要为了全绿而删掉红例。
- 测评 query 不能和 seed 文本同源自循环。

## 6. 学科测验设计

### 6.1 语文

| 题型 | 目标 | 批改方式 |
|---|---|---|
| 生字听写 | 会写重点字词 | 本地严格匹配 |
| 拼音/读音 | 会读生字、多音字 | 本地规则 + 容错 |
| 组词造句 | 理解字词使用 | 规则 + AI 语义判断 |
| 词语解释 | 理解词义 | AI rubric |
| 课文理解 | 抓关键句和主题 | AI rubric + 关键词 |
| 日积月累 | 背诵和应用 | 本地匹配 + AI 判断 |
| 小练笔 | 表达能力 | AI rubric，不直接用唯一答案 |

### 6.2 数学

| 题型 | 目标 | 批改方式 |
|---|---|---|
| 口算/竖式 | 算法熟练 | 本地精确计算 |
| 填空 | 概念掌握 | 本地标准答案 |
| 选择/判断 | 概念辨析 | 本地判分 |
| 应用题 | 审题建模 | 答案精确 + 步骤 AI 诊断 |
| 错因归类 | 找薄弱点 | 小数点、运算顺序、单位、审题、概念 |

### 6.3 英语

| 题型 | 目标 | 批改方式 |
|---|---|---|
| 单词默写 | 拼写准确 | 本地严格匹配 |
| 中英互译 | 词义掌握 | 标准答案 + 同义词 |
| 句型填空 | 语法和搭配 | 本地规则 |
| 连词成句 | 句法 | AI + 关键词 |
| 阅读理解 | 理解能力 | 标准答案 + AI |
| 听写任务 | 家长读/音频读，孩子写 | 本地拼写判分 |

### 6.4 防低级题策略

- 题干和选项不得互相泄露答案。
- 英语题不得上面问 library，下面选项直接出现中文“图书馆”作为明显答案。
- 数学题必须有明确条件、单位和答案。
- 语文开放题必须有评分 rubric，不能要求唯一答案。
- 每套题至少包含基础题、理解题、应用题。

## 7. 学习状态机

### 7.1 任务状态

```text
pending -> in_progress -> paused -> in_progress
in_progress -> stuck -> in_progress
in_progress -> checking -> passed -> completed
checking -> failed -> review_required -> in_progress/checking
completed -> archived
```

### 7.2 按钮规则

| 按钮 | 前置状态 | 动作 | 后置状态 | 幂等要求 |
|---|---|---|---|---|
| 开始 | pending/paused/stuck | 启动或恢复计时 | in_progress | 重复点不重复创建事件 |
| 暂停 | in_progress | 停止计时 | paused | 重复点不增加时间 |
| 我卡住了 | in_progress/paused | 记录卡点并请求辅导 | stuck | 同一卡点可追加 trace |
| 我学会了/继续 | stuck | 回到学习 | in_progress | 不重置任务 |
| 我做完了，开始检查 | in_progress/paused | 生成测验 | checking | 重复点返回已有测验 |
| 提交检查 | checking | 批改并诊断 | passed/failed | 重复提交可覆盖但留 trace |
| 开始下一个任务 | completed/passed | 切换到下一个 pending 任务 | 下一个任务 pending/in_progress | 不回到已完成任务 |

## 8. 时间与排程

### 8.1 每日节奏

默认适合 11 岁孩子暑假使用：

- 上午 09:00-09:35：数学高专注任务。
- 上午 09:45-10:20：语文课文/字词。
- 下午 15:00-15:25：英语单词/句型。
- 下午 15:35-16:00：错题补漏/复习。
- 晚上 19:30-19:45：家长抽查和听写。

### 8.2 排程原则

- 高认知负荷任务放上午。
- 机械记忆和听写放下午或晚上。
- 当天新学和错题补漏比例建议 `7:3`。
- 连续失败的知识点第二天自动提升优先级。
- 每 25-35 分钟必须有休息。

## 9. 数据与记忆

### 9.1 核心数据

- `students`：孩子基本信息。
- `task_sources`：学习来源和长期计划。
- `daily_tasks`：每日任务。
- `task_events`：开始、暂停、卡住、完成等事件。
- `quiz_items`：题目、答案、评分规则。
- `quiz_submissions`：孩子答案和批改结果。
- `skill_mastery`：知识点掌握度。
- `review_queue`：补漏和间隔复习。
- `agent_runs`：Agent 决策记录。
- `agent_trace_steps`：工具调用轨迹。

### 9.2 记忆更新

- 测验分数影响掌握度。
- 错因影响明日补漏优先级。
- 卡住次数影响学习步骤颗粒度。
- 完成时长影响后续任务时间估计。
- 连续通过后降低复习频率，连续失败后提升复习频率。

## 10. 安全与隐私

- `.env` 不进 Git。
- API Key 不输出到页面、日志和报告。
- 不全盘扫描用户电脑找 key。
- Prompt injection 文本不能改写系统规则。
- 孩子端不展示家长/管理能力。
- 数据默认本地 SQLite，换电脑需备份 `.env` 和 `learning.db`。
- 对外部署时必须使用自己的模型 key，并确认公司合规要求。

### 11.2 必测场景

| 场景 | 验收 |
|---|---|
| 管理端发布计划 -> 生成今日任务 -> 孩子端看到 | 任务出现在孩子端，非空且有学习步骤 |
| 孩子点击开始 -> 暂停 -> 再开始 | 计时正常，状态不乱 |
| 孩子点击卡住 | 辅导针对卡点，不直接给答案 |
| 卡住后继续 | 回到当前任务，不重置、不跳错 |
| 完成检查 | 生成测验，重复点击不重复污染 |
| 提交答案 | 得分、错因、掌握度更新 |
| 未通过 | 进入补漏/复习策略 |
| 通过后下一个任务 | 不回到已完成任务 |
| 家长端日报 | 显示结论、风险、建议 |
| 管理端日志 | 能看到 Agent run 和 trace |
| AI 不可用 | 本地规则兜底可用 |
| 资料不足 | 管理端提示覆盖缺口 |

### 11.3 Agent Eval 质量原则

- 不能追求 100% 全绿。
- Known Gap 应保留，代表系统已知边界。
- 数据集不能和实现 seed 同源自循环。
- RAG case 要用真实中文考点。
- Quiz case 要检查答案泄露、题型质量、适龄性。
- Red Team case 要覆盖密钥泄露、prompt injection、越权、直接给答案。

## 13. 面试讲解口径

### 13.1 可以这么讲

这是一个面向儿童学习的受控 Agent 项目。我没有把它包装成完全自主 Agent，而是把 Agent 的不确定性放进受控边界里：家长定义目标，系统基于教材知识库和历史表现生成任务，孩子执行过程通过状态机记录，AI 参与规划、卡住辅导、出题、批改和报告，但所有关键动作都有工具契约、trace、测试和兜底规则。

### 13.2 亮点

- 产品闭环完整：计划 -> 学习 -> 卡住 -> 测验 -> 批改 -> 补漏 -> 家长报告。
- Agent 设计诚实：受控 Runtime，而不是夸大成全自主。
- 测试工程化：功能、UI、并发、Agent Eval、CI、Known Gap。
- 儿童场景安全：不直接喂答案、不泄露密钥、不越权。
- 可扩展：后续可接向量 RAG、真实教材全量切片、LLM-as-Judge。

### 13.3 仍需诚实说明的差距

- 还不是完整 ReAct Agent Loop。
- RAG 仍以关键词为主，向量检索待升级。
- 真实教材资料还需要持续补齐。
- UI 截图审计主要验证“不坏”，不能完全证明“好看”。
- LLM-as-Judge 轨迹评分还可以继续增强。

## 14. 阶段路线图

### 阶段 1：当前可用稳定版

目标：保证三端核心功能能用，测试全绿。

- 修复乱码和实体编码。
- 保证孩子端状态机正确。
- 保证计划、任务、卡住、测验、批改、报告跑通。
- Eval 支持 Known Gap。
- SQLite 支持基础并发。

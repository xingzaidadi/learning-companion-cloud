# AI 测试开发面试问答

## 1. 这是 Agent 还是普通工作流？

这是轻量“受控自主 Agent”。它不是完全自主 Agent，因为学习场景需要家长设目标、孩子确认动作、系统可解释。但它已经具备 Agent 的关键工程特征：目标拆解、工具调用、RAG 检索、记忆沉淀、状态流转、轨迹记录和测评闭环。

## 2. 为什么不用 LangChain？

当前项目选择轻量自研编排，原因是：

- 功能边界清楚，原生 Python 更容易解释和测试。
- 面试展示时能讲清楚每个工具、状态和评估点。
- 避免引入复杂框架后，问题变成框架调参而不是 Agent 工程能力。

如果后续要接多工具、多模型、多 Agent 协作，可以把 `agent_tool_registry.py` 映射到 LangChain Tools、OpenAI function calling 或 MCP。

## 3. Agent 怎么测试？

分 5 层：

- 单点 API 测试：验证端点状态码和返回结构。
- 端到端流程测试：从管理端生成任务到孩子端完成测验。
- 轨迹测试：验证 `agent_trace_steps` 有必要工具和安全决策。
- Golden Set：固定 89 条用例，防止 RAG、计划、卡住、测验能力退化。
- 安全红队：Prompt Injection、密钥泄露、副作用阻断、伪造成绩。

## 4. RAG 怎么测试？

不只看“有没有回答”，而是看：

- 是否检索到资料 chunk。
- 是否带来源 `source_ref`。
- 是否命中预期知识点。
- 出题是否避免答案泄露。
- 导入恶意资料是否不会覆盖系统规则。

## 5. 小测怎么避免低质量？

系统从任务、资料、知识点和学科规则生成题目，并记录质量分。题目返回给孩子端时隐藏答案和解析，批改后才给诊断。英语避免上下文泄露答案，语文覆盖生字词、理解、积累、表达，数学覆盖计算、概念、应用、验算。

## 6. 为什么要记录 Trace？

Agent 最大风险是“看起来能用，但不知道怎么得出结果”。Trace 可以用于：

- 复盘一次计划/卡住/出题的过程。
- 做轨迹级评估。
- 定位工具调用错误。
- 证明系统没有调用危险工具或泄露敏感信息。

## 7. CI 的价值是什么？

普通测试只能证明接口没挂，Agent Eval 能证明智能行为没退化。比如某次改 prompt 或改资料检索后，如果 RAG 命中率下降、卡住提示变成废话、测验泄露答案，CI 会直接失败。

## 8. 这个项目离生产级还有什么差距？

- 用户体系仍是单家庭轻量账号。
- RAG 目前以本地资料为主，版权资料需要家长合法提供。
- LLM-as-Judge 还可以增强，但不能替代规则门禁。
- 完全自主多 Agent 协作不是当前重点，因为孩子学习场景更需要可控。

## 9. 面试怎么 5 分钟演示？

按 `docs/DEMO_SCRIPT_5_MIN.md`：

1. 管理端输入学习目标。
2. 生成今日任务和时间安排。
3. 孩子端开始任务、卡住、获得针对性提示。
4. 完成并进入小测。
5. 查看报告、Trace 和 Eval 报告。

## 10. 如何体现测试开发能力？

这个项目把测试能力从“写脚本点接口”升级到“定义 Agent 行为质量标准”。重点不是测 UI 有没有按钮，而是测 Agent 是否真的能围绕学习结果做正确决策，并且这些决策可回放、可量化、可持续回归。
